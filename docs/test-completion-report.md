## 1. 概要

ISTQB のテスト設計技法（同値分割・境界値分析・デシジョンテーブル・状態遷移・ペアワイズ）を
純粋ロジック 4 モジュールに適用し、**503 件のテストを追加**した（既存 53 件は無変更、合計 556 件・全件成功）。

**本作業で製品ロジックは 1 行も変更していない**（src/ の差分は `APP_VERSION` のバージョン表記 1 行のみで、
本レポートの測定はすべてその変更前のコードに対して行った）。したがって製品の欠陥は 1 件も減っていない。得られたものは次の 3 つである。

1. 現行挙動を固定した**回帰の土台**
2. 8 件の疑義の**可視化**（未修正。修正方針は §7）
3. 境界検出力の**実測**（ミューテーション調査 29 体）

検出力は既存 53 件だけでは 28 体中 10 体（35.7%）だったものが、追加後は 27 体（**96.4%**）になった。
一方で、テストが 1 件も存在しない領域が src の 77% を占めており（§6）、
CI はテストを実行していない（§4）。**「品質を保証した」とは言えない**状態である。

## 5. テスト有効性の実測（ミューテーション調査）

「テストが何件あるか」は品質の指標にならないため、意図的にバグ（ミュータント）を注入し、
テストが検出できるかを測った。

### 手法と限界

- 対象 4 ファイルに **手作業で選定した 29 体**を 1 体ずつ注入し、`npx vitest run` で検出可否を記録
- 各体は「注入 → 実行 → `git checkout` で即復元」をアトミックに実施（同時注入なし）
- **限界**: ツール（Stryker 等）による網羅的ミューテーションではない。
  テストを書いた本人がミュータントを選んでいるため**選択バイアスがある**。
  未テストの `notifications.js` や UI 層には 1 体も注入していない。
  一般的なミューテーションスコアと同列に比較してはならない。

### 結果

| 指標 | 値 |
|---|---:|
| 注入 | 29 体 |
| 検出（killed） | 27 体 |
| 生存（survived） | 2 体 |
| 等価ミュータント（検出不能） | 1 体（M16） |
| **検出率** | **27 / 28 = 96.4%** |
| うち既存 53 件でも検出できた | 10 体 |
| **新規テストが初めて検出した** | **17 体** |

**既存 53 件のみの検出率は 10 / 28 = 35.7%**。追加により 35.7% → 96.4% に上がった。

既存テストが単独で殺せたのは「定数の変更」「分母の誤り」「初期値の誤り」といった
広範囲に波及する粗いバグに限られ、境界条件系（状態境界 M03〜M05、統計境界 M22〜M24、
時刻比較 M14〜M18、ストアのガード M27〜M29）は **1 体も検出できていない**。
BVA とデシジョンテーブルの追加がそのまま検出力になっている。

> **等価判定の訂正経緯**: 検証の初期報告は検出率を 96.3%（26/27、M06 と M16 を等価と見なす）
> としていたが、再検証で **M06 は等価ではない**（テストで殺せる）ことが判明した。
> そこで M06 を殺すテストを追加し、再測定して 27/28 = 96.4% となった。
> 等価判定は「反例が見つからなかった」ではなく「**探索範囲を明示したうえで**見つからなかった」
> と書くべきで、当初の判定は反例が存在する領域を最初から範囲外に置いていた。

### 生存した 2 体

| ID | 箇所 | 判定 | 内容 |
|---|---|---|---|
| **M01** | `date.js:20` `Math.round` → `Math.floor` | **テスト不足（未対応）** | DST のないタイムゾーンでは両者が完全に一致するため検出できない。`TZ=America/New_York` で総当たりすると 4,800 ケース中 **423 件で差異**が出る。`daysBetween` のコメントが明示する「DST は丸めで吸収」という仕様が**一度も検証されていない** |
| **M16** | `notify-plan.js:59` `>` → `>=` | **等価** | 両辺が一致する場合にしか差が出ず、**この比較が評価されるのは `primaryAt > now` が真の分岐の内側だけ**で、そこでは `renudgeAt` が `primaryAt` のちょうど 3 日後（両方 20:00 固定）なので一致し得ない。同じ比較でも `notify-plan.js:75` の else-if 側は等価ではない（`:75` を `>=` にすると 20:00 ちょうどの BVA が落ちる）。`renudgeAt` が Invalid Date のときも `NaN > x` と `NaN >= x` は共に false で同値。**ただし `if` 文自体は生きている**（下記） |

### 対応済みの生存体（M06）

`date.js:59` の `clamp(elapsed / interval, 0, 1)` → `0, 2`。
当初「有効入力域では到達不能な等価ミュータント」と判定したが誤りで、
`intervalDays = -1` かつ `lastReset` が未来のとき `fill` が 1 と 2 で区別できる。
`date.ep-bva.test.js` に該当ケースを追加し、**検出できることを再実行で確認した**
（このテストは Findings #4 の「負の `intervalDays` は常に overdue」という
過大な主張を訂正するテストでもある）。

### M16 に関する重要な訂正 — 「デッドブランチ」は誤り

当初の設計書は `notify-plan.js:59` の内側 `if` を「常に真＝削除可能な防御的コード」と記述していた。
**これは誤りである。** 二分探索で境界を特定し、実測で反例を確認した
（`lastReset = 2026-06-01`、`now = 2026-06-10 10:00`）:

```
interval=99979392 → 予定日 275760-09-10 → ["primary","renudge"]  ← 境界の内側
interval=99979393 → 予定日 275760-09-11 → ["primary"]            ← ★R6 成立の最小値
interval=99979394 → 予定日 275760-09-12 → ["primary"]            ← 窓の内側
interval=99979395 → 予定日 275760-09-13 → ["primary"]            ← 窓の最大値
interval=99979396 → 予定日 Invalid      → []                     ← 予定日ごと表現不能
```

`renudgeAt`（予定日+2日）が先に Date の表現上限を超え、`primaryAt`（予定日-1日）はまだ有効、
という **3 日ぶんの窓**が存在する。

`renudgeAt` が Invalid Date になると `NaN > now` が false になり、内側 `if` は**実際に偽になる**。
つまりこの `if` は「Invalid Date の通知を作らない」という**実働するガード**である。

「デッドだから削除」と判断すると、[`notifications.js:77`](../src/lib/notifications.js#L77) の
`schedule: { at: n.at, allowWhileIdle: true }` に Invalid Date が渡り、例外が同 :81 の catch に落ちて握り潰される。
直前の :64 で既存の予約を `cancel` 済みのため、**primary を含む通知計画全体が登録されないまま、
警告も出ずにリマインダーが全滅する**。

M16（`>` → `>=`）が等価であることと、`if` 文自体が生きていることは矛盾しない。
前者は比較演算子の話、後者は分岐の存在の話である。

**訂正は適用済み**: 設計書 §2-1 の「R6 は幾何的に到達不能」と Findings #6 を訂正し、
上記 5 つの境界値と「この `if` は削除禁止」を `notify-plan.dt.test.js` §2-4 のテストとして固定した。

## 6. 検出した疑義（8 件）

設計中に見つけた 8 件について、独立したエージェントが**反証を試みる形で**個別に検証した
（再現の実行、呼び出し元の全走査、他層のガード探索、深刻度の再判定）。

### 到達可能性の定義

| 段階 | 意味 |
|---|---|
| **L1 通常操作** | 通常の UI 操作から到達する |
| **L2 異常入力** | UI から到達するが、意図的な極端値の入力が必要 |
| **L3 外部データ** | 手編集バックアップの取り込み、または localStorage の改変が必要 |
| **L4 将来変更** | 将来のコード変更で初めて到達する |

### 一覧（深刻度は検証後の値）

| # | 内容 | 判定 | 到達 | 深刻度 | 当初評価 |
|---|---|---|:--:|:--:|---|
| 1 | 未知の将来バージョン（`version:4`）のデータが初期化され、起動時 `persist()` で復元不能になる | CONFIRMED | L4 | **中** | 高 → 中 |
| 8 | `importJSON` が `looksLikeData` を通した入力を `migrate` が捨て、`ok:true`（「バックアップを復元しました」）のままデータが消える | CONFIRMED | L3 | 低 | 中 → 低 |
| 2 | `dateInputToISO` の不正入力が RangeError を投げる | CONFIRMED | L3 | 低 | 中 → 低 |
| 3 | `editHistory` が日付形式を検証せず `lastReset` を汚染する | 部分的 | L3 | 低 | 中 → 低 |
| 4 | 負の `intervalDays` が `normalize` を素通りする | 部分的 | L3 | 低 | 中 → 低 |
| 5 | 空白のみの自転車名が trim されない | 部分的 | L3 | 低 | 低 → 低 |
| 6 | `notify-plan.js:59` の内側 `if`（**デッドブランチではなく実働ガード**） | 部分的 | L2 | 情報 | 低 → 情報 |
| 7 | `totalCount` が date 欠損要素も数える | 部分的 | 到達不能 | 情報 | 低 → 情報 |

**CONFIRMED 3 / 部分的に確認 5 / 反証 0。通常操作（L1）から到達するものは 0 件。**

### 検証で判明した主な訂正

- **#1 は「高」ではなく「中」** — `VERSION` は 2→3 しか出荷されておらず、`version:4` を書くコードは
  一度も存在しない（被害者 0 人）。Android はダウングレードインストールが拒否され、
  アンインストール経由なら WebView のデータごと消えるため、主張したシナリオの半分は成立しない。
  ただし Web/PWA は `registerType:'autoUpdate'` のため、デプロイのロールバックや
  古い SW キャッシュのクライアントという形で**成立し得る**ので「低」までは下げられない。
- **#5 の「実運用では到達不能」は誤り** — `importJSON` 経由で実際に到達する。
  さらに真の欠陥箇所は `notify-plan.js:46` ではなく **`store.js:131` の `normalize`** であり、
  症状は通知本文だけでなくホーム画面・自転車リスト・削除確認ダイアログにも及ぶ。指摘場所を 1 層取り違えていた。
- **#7 は到達不能** — 外部データ経由も `normalizeHistory` が閉じている。「低」ではなく「情報」。
- **#2 は単独では低いが #3 と連鎖する** — `HistoryScreen.jsx:62` が Invalid Date に対し
  `toDateInputValue` から **truthy な `"NaN-NaN-NaN"` を受け取る**ため、`editValue &&` ガードを
  素通りして RangeError に到達する。エラーバウンダリは src 内に 0 件だが、React の
  イベントハンドラ例外はツリーを落とさないため、症状は「保存ボタンが無反応」に留まる。
- **#8 は条件付きで昇格する** — 現行の本番ビルドには plan を pro にする経路が無いため
  import ボタン自体に到達できず「低」。**Pro 課金導線が配線された時点、または `VERSION` を 4 に
  上げた時点で「中〜高」に再評価**すること。「新端末で書き出したバックアップを旧端末で復元」が
  現実的なシナリオになり、最も信頼される操作（バックアップ復元）で成功表示とともにデータを失う。

### 根本原因は 3 系統

8 件という件数は影響度と修正コストの両方を誤って伝える。実際には次の 3 系統に集約される。

| 系統 | 該当 | 根本原因 |
|---|---|---|
| **A** | #1, #8 | `migrate` のバージョン厳密一致で `makeDefaultState()` に落ち、`commit`／起動時 `persist()` が元データを上書きする |
| **B** | #2, #3, #4, #5, #7 | `normalize` / `normalizeHistory` / `pump` の値検証欠如 |
| **C** | #6 | Date の表現上限 |

**系統 B は `store.js` の `normalize` 周辺に数行のバリデータを入れると 5 件が同時に閉じる。**

## 7. 残存リスク

テスト対象外の領域から、影響度の高いものを挙げる。すべて**今回のテストでは検出できない**。

| リスク | 可能性 | 影響 | 根拠 |
|---|:--:|:--:|---|
| **CI がテストを実行せず、リグレッションを検出できない** | 高 | 高 | `deploy.yml` は `npm ci` + `npm run build` のみ。テスト全滅でもデプロイ成功 |
| **Pro では複数台を追加できないのに UI が「Proで解放されます」と案内する** | 高 | 高 | `store.js:19` は `pro.bikes = 1`（`Infinity` は premium のみ）。一方 UI の文言は `BikeSheet.jsx:84`「複数の自転車の管理はProで解放されます」、`SettingsScreen.jsx:65`「複数の自転車はProで解放されます」。`BikeSheet.jsx:12` に「⚠ ペイウォール実装時に要再整合」と自己申告のコメントあり |
| **Web/PWA ユーザーはリマインダーを 1 件も受け取れない** | 高 | 高 | `notifications.js:12` で非 native は全関数が早期 return。Web Notification / Push へのフォールバックがコード中に存在しない。仕様か未実装かが未決 |
| **通知の実登録層が完全に未検証で、CI では永久に検出できない** | 中 | 高 | `notifications.js` 92 行が未テスト。jsdom では `NATIVE=false` で全関数が即 return。失敗は :83 の `console.warn` で握り潰される |

## 9. プロセスの評価と教訓

### 1. 自己レビューは到達可能性を系統的に過大評価する

8 件の疑義の深刻度は、独立検証によって **8 件中 7 件が下方修正**された（#5 のみ据え置き。上方修正は 0 件）。

```
#1 高→中  #2 中→低  #3 中→低  #4 中→低  #8 中→低  #6 低→情報  #7 低→情報
```

設計者本人は「コードにこう書いてある」から欠陥を導くため、
**呼び出し元のガードや UI の入力制約を数え落としやすい**。
実際、#2 は `<input type="date">` の value sanitization、#4 は書き込み経路 4 箇所すべての
`>= 1` ガード、#7 は `normalizeHistory` によって、いずれも到達性が大きく狭まっていた。
欠陥報告に**到達経路を必ず書かせる**運用にすれば、この過大評価は設計時点で潰せる。

### 2. 「デッドコード」の判定は総当たりの探索範囲で決まる

設計時に `notify-plan.js:59` を「幾何的に到達不能」と結論したが、その根拠にした総当たりは
`interval ∈ {7,14,21,28} × 予定日±30日` という**現実的な入力域に限定されていた**。
Date の表現上限（西暦 275760 年）を範囲に入れた途端に反例が出る。
**「反例が見つからなかった」ではなく「この範囲では見つからなかった」と書く**べきだった。

### 3. 件数はテストの価値を表さない

556 件という数字は、それ単体では何も証明しない。
価値を実際に示したのはミューテーション調査で、
**既存 53 件が境界条件系のミュータントを 1 体も殺せていなかった**という事実が、
BVA とデシジョンテーブルを追加した意味を初めて定量化した。
今後は「件数」ではなく「検出率」を報告指標にする。

### 4. スコープ外の判断は根拠を伴わないと穴になる

DST 検証は「日本には DST が無い」という理由でスコープ外にしたが、
その判断はミューテーション M01 の生存という形で**測定可能な穴**として現れた。
スコープ外にすること自体は妥当でも、
**「未検証である」と明記しないと「検証して問題なかった」と読まれる**。
